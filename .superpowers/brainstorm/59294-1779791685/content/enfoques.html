A — HTTP ligero + descarga PDF

Escaneo en el navegador con ZXing-js. El backend usa requests con sesión cacheada para buscar el albarán en el portal y descargarlo como PDF. Para otros labs: OCR.

Ventajas

  • Muy ligero en Render (sin navegador)
  • Rápido (<2s si cookie válida)
  • PDF 100% oficial sin OCR

Riesgo

  • Si Hefame cambia su HTML o añade bot-protection → se rompe
📷 Cámara → ZXing-js (navegador)
↓ número de albarán
🎭 Backend: Playwright headless
→ Navega a Hefame/BIDA real
→ Login visual completo
→ Descarga PDF
─────────────────
❌ Si falla → OCR actual

B — Playwright (navegador real)

Playwright automatiza un navegador Chromium real. Más robusto ante cambios del portal, soporta JS pesado y posibles CAPTCHAs.

Ventajas

  • Más resistente a cambios del portal
  • Funciona en portales JS-heavy

Riesgo

  • ~150MB extra en Render
  • Lento (~5-10s por albarán)
  • Caro en plan gratuito

│ └─ HTML → reconstruir PDF
└─ Otro lab → OCR actual

+ input manual del número

C — Híbrido inteligente ⭐ Recomendado

Como A, pero detecta el proveedor automáticamente por el formato del número de albarán (prefijo Hefame vs BIDA). Si no descarga PDF directo, reconstruye desde HTML. Fallback a OCR para labs externos. Incluye campo manual si el código no se lee bien.

Ventajas

  • Ligero en Render (sin Playwright)
  • Cubre Hefame + BIDA + otros labs
  • Robusto: 3 niveles de fallback
  • No depende de un solo camino

Riesgo

  • Necesita inspeccionar el portal de Hefame una vez para mapear su HTML
Nota sobre el "agente siempre conectado": El enfoque C y A usan un SessionManager en Python — no un proceso extra. La cookie de Hefame/BIDA vive en memoria del servidor. Si Render reinicia (ocurre en plan gratuito), hace login automático al primer escaneo. Latencia: <1s con cookie válida, ~3s si hace re-login.
